### 21.1 目的

NotebookLMのような動的展開型マインドマップUIを活用し、以下を実現する**品質ゲート基盤**を構築する。

- 工程ごとの品質チェック
- 承認ログ記録
- 証跡出力
- フェーズ別テンプレート量産

社内利用に留まらず、将来的なパッケージ販売・商用展開を視野に入れる。

---

### 21.2 基本コンセプト：テンプレート駆動型アプリケーション

UIやロジックを固定化し、以下の要素をすべて変数化する。

- テキスト / input / button
- 表示・非表示制御
- ツリー構造（ノード型UI）
- フェーズ内容

外部定義ファイル（JSON/YAML）とマージすることで、**単一エンジンから複数フェーズのアプリを生成**する。

テンプレートを差し替えるだけで、企画・要件定義・基本設計など各フェーズに対応したチェックリストアプリを量産できる構造とする。

---

### 21.3 対応フェーズ

単一エンジンから以下の全フェーズを対象とするアプリを生成可能とする：

| フェーズ | 内容 |
| -------- | ---- |
| 企画 | 企画段階のQA観点・スコープ・成功条件確認 |
| 要件定義 | 要件妥当性・曖昧語・トレーサビリティ確認 |
| 基本設計 | 構造整合・インタフェース定義確認 |
| 詳細設計 | 詳細仕様・スキーマ準拠確認 |
| 実装 | コードレビュー観点・実装チェック |
| 運用 | 運用保守・障害対応観点確認 |

### 21.4 PWA採用方針

PWA（Progressive Web App）を採用することで以下を実現する：

- ブラウザベース開発（追加インストール不要）
- インストール可能なアプリ形態（デスクトップアイコン対応）
- オフライン利用対応（サービスワーカー活用）
- PCアプリに近いUX
- サーバー非依存構成（Phase 1はローカル完結）

サーバーコストを抑えつつ、将来的なSaaS化も可能な**中間構造**として機能させる。

### 21.5 認証・ロール設計の基本方針

#### ロール定義

品質ゲート運用における2つの基本ロールを定義する：

| ロール | 主な役割 | 導入フェーズ |
| ------ | -------- | ------------ |
| User | チェックリストの実行・状態更新・JSONエクスポート | Phase 1〜 |
| Admin | User権限 + 全ユーザーログ閲覧・ユーザー/ロール管理・管理者追加 | Phase 2〜 |

#### 認証機能の段階導入

| フェーズ | 認証方式 | 特徴 |
| -------- | -------- | ---- |
| Phase 1（MVP） | 認証なし（任意） | 確認者名を自己申告入力、ローカル完結 |
| Phase 2（証跡強化） | ID/Password認証 | ロール別アクセス制御、管理者ダッシュボード導入 |
| Phase 3（商用化） | 高度な認証（SSO等） | 組織階層対応、監査対応レベルの認証 |

#### 管理者機能の概要（Phase 2以降）

管理者（Admin）が実施可能な主要機能：

- **ユーザー管理**: 新規ユーザー作成、初期パスワード発行、ロール指定
- **ロール管理**: 既存ユーザーの権限変更（User ⇔ Admin）
- **ログ閲覧**: 全ユーザーの承認履歴を検索・フィルタ・エクスポート可能

> **→ 詳細な権限境界・UI設計は基本設計フェーズで確定する**

### 21.6 UI機能の基本方針

#### ノード出現制御

- 初期状態では最初のチェックリスト項目（ノード）のみ表示
- 前のノードが`Done`または`Pending`になると、動的に矢印が伸びて**次ノードが1つずつ出現**
- NotebookLM風の段階的展開により、進行状況を視覚的に把握可能

#### サイドパネルの詳細機能

ノードクリック時に画面右側に展開するサイドパネルには以下を含める：

**入力項目**
- 実施者名（必須）
- 詳細チェック項目（チェックボックスリスト、テンプレートで定義）

**表示項目**
- 説明文・検証方法・合格基準
- PASS/FAIL/ABORT例（判断の参考情報）

**選択項目**
- **リスクレベル**: S / A / B / C のドロップダウン（テンプレートでデフォルト値設定可能）
- **ステータス**: ToDo / InProgress / Pending / Done
- **Pending理由**: Pending選択時のみ表示・入力必須

#### カスタム項目追加機能

- すべての定義済みチェックリスト項目が完了した時点で`＋ボタン`を表示
- ユーザーが独自のチェック項目を最大5件まで追加可能
- カスタム項目も既存ノードと同じUI・操作で扱える
- プロジェクト固有の確認事項に柔軟に対応

#### JSONエクスポート仕様

- **配置**: フェーズ画面の左上に常時固定表示
- **出力条件**: 任意時点で出力可能（InProgress状態でも可）
- **想定用途**: CI連携テスト用の途中状態確認、レビュー・監査時の証跡提出
- **出力内容**: 現在状態のスナップショット + ログ履歴 + テンプレートID/バージョン

> **→ 詳細なJSON構造・スキーマは詳細設計フェーズで確定する**

### 21.8 開発フェーズ計画

#### Phase 1：MVP（動作確認版）

**目的**：基盤エンジンの確立

| 項目 | 内容 |
| ---- | ---- |
| 構成 | Next.js + PWA + ローカル保存（localStorage / IndexedDB） |
| 出力 | JSONエクスポート / ログ出力機能 |
| 特徴 | サーバーコストゼロ・単体利用可能・品質ゲートUI完成 |

「まず動くもの」を構築し、内部での視座共有を実現する。

#### Phase 2：証跡強化版

**目的**：品質保証レベルの向上 + 認証・管理機能の導入

| 追加機能 | 内容 |
| -------- | ---- |
| 認証機能 | ID/Password認証、ロール別アクセス制御 |
| 管理者ダッシュボード | ユーザー管理、ロール管理、全ユーザーログ閲覧 |
| ユーザー名入力必須 | 承認者情報を全ログに付与 |
| ログ署名ハッシュ生成 | JSONエクスポート時に改ざん検知を可能にする |

効果：改ざん検知可能なログ構造 / 承認責任の明確化 / 組織内での役割分担が可能。ローカル時刻を使用。

> **→ ハッシュ生成アルゴリズムの詳細は詳細設計フェーズで定義する**

#### Phase 3：商用化対応

**目的**：パッケージ販売・外部展開

| 追加構成 | 内容 |
| -------- | ---- |
| 高度な認証 | SSO等の企業向け認証方式 |
| サーバー保存 | 承認履歴のDB管理 |
| サーバー時刻取得 | ローカル時刻改ざんリスクを排除 |
| 電子署名 | 本人性の証明（監査対応） |
| 組織階層対応 | 部門・チーム単位のアクセス制御 |

効果：監査対応レベルの証跡 / チーム共有 / クラウド型提供が可能になる。

> **→ 認証方式・DB設計の詳細は要件定義〜設計フェーズで定義する**

### 21.9 ログ証跡レベル設計

段階的拡張を前提とした3段階の証跡強度を定義する。

| レベル | 保存方法 | 証跡強度 | 対応フェーズ |
| ------ | -------- | -------- | ------------ |
| Lv1 | ローカル保存（IndexedDB） | 低 | Phase 1（MVP） |
| Lv2 | 署名付きJSON（ハッシュ検証） | 中 | Phase 2（証跡強化） |
| Lv3 | サーバー保存 + サーバー時刻 | 高 | Phase 3（商用化） |

> **→ 各レベルのフォーマット仕様・ハッシュ計算方法は詳細設計フェーズで定義する**

### 21.10 Gate Runnerとの並行開発（依存関係）

本CFUIは、Gate Runner（Dockerfile自動生成基盤）と並列開発が必要である。

**理由**：

- G2工程にてCheckFlowが生成する `checklistresults.json` が、Gate Runnerの入力として必須
- CheckFlowの出力JSONをGate Runnerが消費・検証する構造であるため、**データフォーマットの早期確定が両プロジェクトに影響する**

**必要なアクション**：

- JSON定義仕様の早期確定（企画完了後、要件定義フェーズ冒頭で実施）
- 両プロジェクト間のデータフォーマット整合確認

> **→ `checklistresults.json` スキーマの確定は要件定義フェーズで最優先対応とする**

### 21.11 商業展開の可能性

本基盤は以下のいずれのビジネスモデルにも転用可能な構造を持つ：

| モデル | 概要 |
| ------ | ---- |
| 社内展開 | 既存チームへの段階導入（Phase 1〜2で実現） |
| パッケージ販売 | フェーズ別テンプレートとエンジンのセット販売 |
| SaaS展開 | クラウド型提供（Phase 3以降） |
| 人月型ビジネス | 導入支援・テンプレートカスタマイズ |

特に「**最終判断者の承認証跡管理**」を強みとし、品質ゲート基盤としての差別化を図る。

### 21.12 リスクと対策

| リスク | 対策 | 対応フェーズ |
| ------ | ---- | ------------ |
| ローカル改ざん | Phase 2以降で署名ハッシュを導入 | Phase 2 |
| 証跡強度不足（監査対応） | 商用版でサーバー時刻採用 | Phase 3 |
| 開発肥大化 | 段階的フェーズ開発で管理 | 全フェーズ |
| JSON仕様の不整合 | Gate Runnerとの早期仕様合意 | 要件定義 |

### 21.13 後工程への引継ぎ観点

企画段階では方針を定め、以下の詳細は各フェーズで確定する。

#### 【要件定義フェーズへの引継ぎ】

- **ロール設計の詳細**：Admin（全機能 + ユーザー管理）/ User（チェック実行のみ）の権限境界を定義する
- **認証要件**：Phase 2向けID/Password認証の詳細仕様、Phase 3向けSSO等の選定
- **ステータス種別と遷移ルール**：ToDo / InProgress / Pending / Done の制御仕様（Pending時の理由必須ロジックなど）
- **リスクレベル定義**：S / A / B / C の評価基準、デフォルト値設定ルール
- **JSONエクスポート仕様**：出力条件・CI連携時のFail判定ルール（未完了項目あり = FAIL）
- **カスタム項目追加機能**：最大5件の制限根拠、入力可能項目の詳細仕様
- **`checklistresults.json` スキーマ確定**：Gate Runnerとのインタフェース仕様として最優先で合意する

#### 【基本設計フェーズへの引継ぎ】

- **画面遷移設計**：ログイン → Top → フェーズ画面 → チェックリスト → サイドパネル展開の全フロー
- **状態遷移設計**：ノード単位のステータス遷移と次ノード解放ロジック（1つずつ出現）
- **UIコンポーネント構成**：
  - ツリー構造ノード / サイドパネル / 固定ヘッダ（JSONエクスポートボタン）
  - 詳細チェック項目（チェックボックスリスト）
  - リスクレベル・ステータスのドロップダウン
  - カスタム項目追加の`＋ボタン`とモーダル
- **管理者ダッシュボード設計**：ユーザー一覧・ロール変更・承認履歴閲覧のUI構成
- **アニメーション仕様**：ノード間の線描画・次ノード出現演出（動的に伸びる矢印）

#### 【詳細設計フェーズへの引継ぎ】

- **JSONデータ構造設計**：
  - チェックリスト定義ファイル（テンプレートYAML/JSON）のスキーマ
  - 詳細チェック項目（`detailedChecks`）の配列構造
  - リスクレベル（`riskLevel`）のデフォルト値と選択肢
  - カスタム項目（`isCustom`フラグ）の識別方法
- **ログフォーマット仕様**：`checkedby` / `timestamp` / `status` / `reason` / `riskLevel` / `evidence_refs` の詳細型定義
- **署名ハッシュ生成アルゴリズム**（Phase 2）：入力値・アルゴリズム選定・検証方法
- **バックエンド拡張設計**（Phase 3）：認証API / ログ保存API / DB設計
- **オフライン同期設計**：Service WorkerによるキャッシュとオンラインDB同期の整合
