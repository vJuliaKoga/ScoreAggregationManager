### 9.1 ポリシー

- スコアは合否の唯一根拠にしない（最終判断は人＝CoachのDone/Abort）
- ただし、品質劣化を早期検知するために 0.70未満はWarningを必ず出す
  - Allure上でWarningとして表示
  - "理由要約／改善ガイド／再実行条件"をセットで出す（初学者が萎えないため）

この0.70は、要件妥当性チェックリスト雛形にある `pass_threshold: 0.7` を社内標準の警告ラインとして採用する。

### 9.2 Deep Eval / Promptfooの扱い

- Deep Eval：品質の"定量指標"。モデル変動を前提に、スコア推移を記録し劣化を検知する（Allureに推移も集約）。
- Promptfoo：プロンプト品質の回帰テストとして位置づけ、テンプレ改善サイクルの根拠にする。

### 9.3 評価思想：構造の逸脱とリスク管理

本基準における最優先事項は「構造的整合性」である。パフォーマンスや要件の曖昧さは後工程での最適化が可能だが、構造的欠陥（PageObjectの崩壊、再利用不能な設計）は不可逆的なコストを強いるため、最も厳格に管理される。

#### 品質評価の重要度階層（ウェイト定義）

| 優先順位 | ゲート | ウェイト | 判定の視点 |
| -------- | ------ | -------- | ---------- |
| 1 | G2 Checklist / 人による判断 | **30%** | 実運用の文脈における人間の直感と経験を最重視する。実運用者の判断ログは最も信頼に足る品質指標。 |
| 2 | G3 Structure Lint / 構造チェック | **25%** | 「将来コスト」に直結する。設計の崩壊を許せば顧客信頼度の維持は不可能になるため、極めて高いウェイトを配分。 |
| 3 | G4 AI Evaluation / DeepEval | **20%** | 「判断材料の提示」を主目的とする。生成物の整合性を評価し、壊滅的な劣化を防ぐ防波堤として機能させる。 |
| 4 | PF Promptfoo / プロンプト評価 | **15%** | モデル変更や調整時の品質変化をトラッキングする定量評価。 |
| 5 | G1 Ambiguity Check / 曖昧語チェック | **10%** | 設計崩壊の予兆。初期スクリーニングとしての役割であり、後工程での吸収が可能なため相対的ウェイトは低く設定。 |

### 9.4 二段階判定モデル

品質判定は、致命的な欠陥を排除する「第一段階」と、定量的健全性を測定する「第二段階」で構成される。

#### 第一段階：致命条件（Fatal Conditions）

以下の条件のいずれかに抵触した場合、総合スコアに関わらず **即時失格（Instant Fail）** とする。

| 条件 | 対象ゲート | 基準 |
| ---- | ---------- | ---- |
| CI失敗 | — | パイプライン自体の異常終了 |
| トレーサビリティ欠落 | G5 | ID参照の不整合、または要件に紐付かない「孤立した定義」の存在 |
| 構造重大違反 | G3 | スキーマ定義の逸脱など、設計の根幹を損なう違反 |
| Faithfulness（整合性）壊滅 | G4 | DeepEvalスコアが **0.6未満** |
| 曖昧語の過剰検出 | G1 | 検出数が **10件超**（設計崩壊の前兆とみなす） |

#### 第二段階：重み付き合算スコア（計算式）

第一段階を通過した案件に対し、以下の数式を用いて最終スコアを算出する。

```
Total Score = (G1 × 0.10) + (G3 × 0.25) + (G4 × 0.20) + (PF × 0.15) + (G2 × 0.30)
```

### 9.5 ガバナンス指針：案件別リスクレベルの設定

「品質」は固定的な絶対値ではなく、案件の特性とリスク許容度に応じて定義される。

| リスクレベル | 合格必須スコア | 運用指針 |
| ------------ | -------------- | -------- |
| LOW Risk | **70点以上** | 軽微な構造違反や数件の曖昧語は減点対応とし、スピードを優先。 |
| HIGH Risk | **90点以上** | 厳格な統制を適用。微細な逸脱も修正対象とし、設計の純粋性を担保。 |

### 9.6 G2（Checklist）のCI統合要件

人による判断をシステム的に保証するため、CI接続時に以下のロジックを強制する。

- **Fail（ビルド停止）条件**：
  - 項目を「Abort（中断）」した際、その理由（正当な根拠）がログに記載されていない
- **Warning（警告）条件**：
  - Abort率が規定値を超え、「判断の回避」が常態化している
- **内部重み付け**：
  - G2内でも、ドメインやフェーズに応じて項目ごとに内部重みを設定し、重要な判断の漏れをスコアに直結させる
