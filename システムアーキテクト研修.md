# システムアーキテクト研修資料
**対象：初級自動化エンジニア（入社1ヶ月～）**  
**目標：上級自動化エンジニア認定に必要なアーキテクトスキルの習得**

---

## 1. システムアーキテクトってなに？

### 1.1 基本的な役割

一言でいうと、**「自動化テストの設計図を描く建築家」**です。

プランナーが「何を自動化するか」を決め、エンジニアが「実際に作る」とすれば、アーキテクトは「どう設計するか」を決めます。

**3つの役割の違い：**

| 役割 | プランナー（軍師） | アーキテクト（建築家） | エンジニア（職人） |
|------|-------------------|---------------------|-------------------|
| 関心事 | 何を自動化すべきか | どう設計すべきか | どう実装するか |
| 質問 | 「このテスト、ROIはある？」 | 「この設計、3年後も保守できる？」 | 「このステップ、動く？」 |
| アウトプット | 自動化計画書 | 設計書・アーキテクチャ図 | 動作するテストコード |
| 視点 | ビジネス価値 | 技術的持続可能性 | 技術的実現可能性 |

### 1.2 なぜアーキテクトが必要なのか？

**よくある失敗例：設計なしで作り始めた結果...**
```
❌ 悪い例：「とりあえず動けばOK！設計なんて後回し！」

【3ヶ月後の惨状】
- テストケースが500件に増えたが、どこに何があるか分からない
- 同じログイン処理が100箇所にコピペされている
- ボタン名が変わっただけで100件のテストが壊れる
- 新メンバーが「何がどうなってるか分からない」と困惑
- メンテナンスに毎週10時間かかる...

✅ 良い例：「まず設計を考えよう」

【設計のある世界】
- フォルダ構成が整理され、目的のテストがすぐ見つかる
- ログイン処理は共通部品化され、1箇所直せば全体に反映
- 画面変更があっても影響範囲が限定的（修正は数箇所のみ）
- 設計書を見れば新メンバーもすぐに理解できる
- メンテナンスは週1時間程度
```

**アーキテクトの思考回路：**

1. 「1年後、テストが1000件になっても管理できる？」（スケーラビリティ）
2. 「画面変更があったとき、影響範囲を最小化できる？」（保守性）
3. 「他のプロジェクトでも使える設計になってる？」（再利用性）
4. 「新メンバーが見ても理解できる？」（可読性）
5. 「3年後も技術的負債にならない？」（持続可能性）

### 1.3 アーキテクトの責任範囲

**アーキテクトが決めること：**

| 項目 | 具体例 |
|------|--------|
| **構成管理** | フォルダ構造、ファイルの配置ルール |
| **命名規則** | テストケース名、変数名、関数名の付け方 |
| **共通部品** | どの処理を共通化し、どう管理するか |
| **データ管理** | テストデータの保存場所、形式 |
| **エラー戦略** | エラー発生時の対応方針 |
| **実行戦略** | いつ、どのテストを、どの順序で実行するか |
| **ツール選定** | 使用するライブラリ、フレームワーク |
| **品質基準** | コードレビューの基準、品質ゲート |

## 2. アーキテクトに必要な知識

### 2.1 設計原則（SOLID原則の自動化版）

ソフトウェア設計の基本原則を、テスト自動化に適用します。

#### 原則1：単一責任の原則（SRP: Single Responsibility Principle）

**1つのテストケースは1つの目的だけを持つべき**
```
❌ 悪い例：「ユーザー登録から購入完了までの全フロー」
   → 1つのTCで20ステップ、どこで失敗したか分かりにくい
   → 修正時に影響範囲が大きい

✅ 良い例：機能ごとに分割
   - TC001：ユーザー登録機能
   - TC002：ログイン機能
   - TC003：商品検索機能
   - TC004：カート追加機能
   - TC005：購入確定機能

   → それぞれ5ステップ程度で完結
   → 失敗箇所が明確、修正も局所的
```

**メリット：**
- 失敗原因の特定が早い
- 修正範囲が限定的
- テストの再利用性が高い

#### 原則2：依存性逆転の原則（DIP: Dependency Inversion Principle）

**具体的な実装ではなく、抽象的なインターフェースに依存すべき**
```
❌ 悪い例：画面の具体的な構造に依存
   XPath: "/html/body/div[3]/form/div[2]/button[1]"
   → 画面のレイアウトが変わると即座に壊れる

✅ 良い例：論理的な意味に依存
   ID指定: "#submit-button"
   data属性: "[data-testid='submit-button']"
   → ボタンの位置が変わっても動作する
```

**MagicPodでの実践例：**
```
【悪い設計】
- ステップ：XPathで要素を直接指定
  → 画面構造の変更に弱い

【良い設計】
- 画面定義：論理名で要素を定義（Page Object）
- ステップ：論理名で操作を記述
  → 画面が変わっても画面定義だけ修正すればOK
```

#### 原則3：開放閉鎖の原則（OCP: Open-Closed Principle）

**拡張に対しては開いていて、修正に対しては閉じているべき**
```
❌ 悪い例：ハードコーディング
   ログインID = "test@example.com"
   → 別のユーザーでテストしたいとき、コードを修正する必要がある

✅ 良い例：外部データ化
   ログインID = CSVファイルから読み込み
   → データファイルを変えるだけで色々なパターンをテスト可能
   → コード自体は一切変更不要
```

**データドリブンテストの実装例：**

| テストデータ（CSV） | テストロジック（固定） |
|-------------------|---------------------|
| user1@example.com, pass123 | 1. CSVから読み込み |
| user2@example.com, pass456 | 2. ログイン実行 |
| user3@example.com, pass789 | 3. 結果確認 |

→ 3パターンのテストがコード変更なしで実行可能

### 2.2 設計パターン

自動化テストでよく使われる設計パターンを理解しましょう。

#### パターン1：Page Object Pattern（画面オブジェクトパターン）

**画面の構造と操作を1箇所にまとめる**

**概念図：**
```
【従来の方式】
TC001 → 画面Aの要素を直接操作
TC002 → 画面Aの要素を直接操作
TC003 → 画面Aの要素を直接操作
↓
画面Aの変更 → 全TCを修正（100箇所）

【Page Object Pattern】
TC001 ┐
TC002 ├→ LoginPage（画面定義） → 画面Aの要素
TC003 ┘
↓
画面Aの変更 → LoginPageだけ修正（1箇所）
```

**具体例（MagicPod風）：**
```
【画面定義：LoginPage】
- email_input: #email
- password_input: #password
- login_button: #submit
- error_message: .alert-error

【操作定義：LoginPageActions】
- login(email, password):
    1. email_inputに入力
    2. password_inputに入力
    3. login_buttonをクリック

【テストケース】
TC001: 正常ログイン
  LoginPage.login("user@example.com", "password123")
  → 結果確認

TC002: エラーログイン
  LoginPage.login("invalid@example.com", "wrong")
  → error_messageの表示確認
```

**メリット：**
- 画面変更時の修正は画面定義のみ
- テストコードが読みやすい（英語の文章のよう）
- 重複コードが減る

#### パターン2：Data-Driven Pattern（データ駆動パターン）

**テストロジックとテストデータを分離**

**構造：**
```
【テストロジック（固定）】        【テストデータ（可変）】
1. データを読み込む           ← users.csv
2. ログインを実行                 - user1, pass1
3. 結果を確認                     - user2, pass2
4. 次のデータへ                   - user3, pass3
```

**実装例：**
```
【データファイル：login_test_data.csv】
test_id, email, password, expected_result
TC001, valid@example.com, pass123, success
TC002, invalid@example.com, wrongpass, error
TC003, empty@example.com, , error
TC004, user@example.com, 12345, error

【テストスクリプト（擬似コード）】
for each row in csv:
    login(row.email, row.password)
    assert result == row.expected_result
```

**メリット：**
- 新しいテストケース追加が簡単（CSVに1行追加するだけ）
- コードを書けない人でもテストデータを作れる
- 大量のパターンを効率的にテスト可能

#### パターン3：Keyword-Driven Pattern（キーワード駆動パターン）

**業務フローをキーワード（命令）で表現**

**概念：**
```
【キーワード定義】
- LOGIN: ログイン処理を実行
- SEARCH_PRODUCT: 商品を検索
- ADD_TO_CART: カートに追加
- CHECKOUT: 購入手続き

【テストシナリオ（Excel/CSV）】
Step1: LOGIN, user@example.com, password123
Step2: SEARCH_PRODUCT, ノートPC
Step3: ADD_TO_CART
Step4: CHECKOUT
```

**メリット：**
- 非エンジニアでもテストシナリオを書ける
- 業務フローが可視化される
- シナリオの再利用性が高い

### 2.3 命名規則の重要性

**良い命名 = 設計品質の指標**

#### テストケース名の命名規則

**推奨フォーマット：**
```
[機能分類]_[テスト目的]_[条件]_[期待結果]

例：
✅ LOGIN_正常系_有効な認証情報_ログイン成功
✅ LOGIN_異常系_無効なパスワード_エラーメッセージ表示
✅ SEARCH_境界値_0件ヒット_結果なしメッセージ表示
✅ CART_追加操作_最大数超過_エラー表示
```

**命名のルール：**

| 項目 | 良い例 | 悪い例 | 理由 |
|------|-------|-------|------|
| **具体性** | LOGIN_異常系_パスワード誤り | test001 | 内容が分からない |
| **一貫性** | SEARCH_正常系_1件ヒット | SearchTest, search1 | 表記揺れがある |
| **検索性** | CART_追加操作_在庫なし | カート関連のテスト | 検索できない |
| **長さ** | CHECKOUT_正常系_クレジット決済 | CHECKOUT_正常系_クレジットカードによる決済処理が正常に完了することを確認するテスト | 長すぎる（80文字以内） |

#### フォルダ構造の設計

**推奨構造（階層型）：**
```
/tests
├── /01_認証機能
│   ├── /01_ログイン
│   │   ├── LOGIN_正常系_標準ユーザー.txt
│   │   ├── LOGIN_異常系_パスワード誤り.txt
│   │   └── LOGIN_境界値_最大文字数.txt
│   ├── /02_ログアウト
│   └── /03_パスワード変更
├── /02_商品機能
│   ├── /01_検索
│   ├── /02_詳細表示
│   └── /03_在庫確認
├── /03_購入機能
│   ├── /01_カート操作
│   ├── /02_決済処理
│   └── /03_注文確認
└── /共通部品
    ├── common_login.txt
    ├── common_logout.txt
    └── common_data_setup.txt
```

**命名のポイント：**
- 番号プレフィックス（01_, 02_）で実行順序を制御
- 機能軸で大分類、操作軸で小分類
- 共通部品は専用フォルダに分離

## 3. 設計の実践テクニック

### 3.1 共通部品化の戦略

**共通化すべきものの判断基準：**

$$共通化すべき = 使用回数 \times 保守頻度 \times ステップ数$$

**判断フローチャート：**
```
この処理は3回以上使われる？
  ↓ YES
この処理は仕様変更の可能性がある？
  ↓ YES
この処理は5ステップ以上ある？
  ↓ YES
→ 共通部品化すべき！

1つでもNOなら → 共通化不要（コピペでOK）
```

**具体例：**

| 処理 | 使用回数 | 保守頻度 | ステップ数 | 判定 |
|------|---------|---------|----------|------|
| ログイン | 100回 | 高 | 5 | ✅ 共通化必須 |
| ログアウト | 50回 | 低 | 2 | ✅ 共通化推奨 |
| 商品検索 | 30回 | 中 | 8 | ✅ 共通化推奨 |
| エラー確認 | 5回 | 低 | 3 | △ 任意 |
| 特殊データ投入 | 1回 | 高 | 10 | ❌ 共通化不要 |

**共通部品の設計例：**
```
【共通部品：common_login】
パラメータ：
  - user_email: ログインID
  - user_password: パスワード

ステップ：
  1. ログイン画面を開く
  2. ${user_email}を入力
  3. ${user_password}を入力
  4. ログインボタンをクリック
  5. ダッシュボード表示を確認

【呼び出し側】
TC001:
  - common_loginを呼び出し(user1@example.com, pass123)
  - 以降のテスト処理...

TC002:
  - common_loginを呼び出し(admin@example.com, adminpass)
  - 以降のテスト処理...
```

**共通化のメリット計算：**
```
【共通化前】
- 100個のTCにログイン処理（5ステップ）をコピペ
- ログイン画面の変更 → 100箇所を修正（500分 = 8.3時間）

【共通化後】
- 100個のTCは共通部品を呼び出すだけ（1ステップ）
- ログイン画面の変更 → 共通部品1箇所を修正（5分）

削減効果 = 8.3時間 - 5分 ≒ 8時間（1人日分の削減！）
```

### 3.2 エラーハンドリング戦略

**エラー発生時の対応を設計段階で決める**

#### エラーの分類と対応方針

| エラー種類 | 原因例 | 対応方針 | 実装例 |
|----------|-------|---------|-------|
| **一時的エラー** | 通信タイムアウト | リトライ | 3回まで再試行 |
| **環境依存エラー** | テストデータ不足 | スキップ | 事前条件チェック |
| **致命的エラー** | 画面構造の大幅変更 | 即座に停止 | 管理者に通知 |
| **期待されるエラー** | バリデーション | 検証 | エラーメッセージ確認 |

#### リトライ戦略の設計
```
【リトライ設計例】

処理: ログインボタンをクリック

1回目失敗
  ↓
待機（5秒）
  ↓
2回目実行
  ↓
失敗
  ↓
待機（10秒）
  ↓
3回目実行
  ↓
それでも失敗 → テスト失敗として記録
```

**MagicPodでの実装イメージ：**
```
ステップ1: ログインボタンをクリック
  - リトライ回数: 3
  - リトライ間隔: 5秒
  - タイムアウト: 30秒

ステップ2: エラー時のスクリーンショット取得
  - 条件: 前ステップが失敗した場合
```

#### エラーログの設計

**記録すべき情報：**
```
【エラーログの必須項目】
1. タイムスタンプ: 2025-03-05 14:23:45
2. テストケースID: LOGIN_異常系_パスワード誤り
3. 失敗ステップ: ステップ3「パスワード入力」
4. エラー内容: 要素が見つかりません（#password）
5. スクリーンショット: screenshot_20250305_142345.png
6. 実行環境: iOS 17.1, Safari
7. 期待値: ログイン失敗メッセージ表示
8. 実際値: 何も表示されず
```

**ログの保存場所設計：**
```
/logs
├── /2025-03
│   ├── /2025-03-05
│   │   ├── test_execution.log（全体ログ）
│   │   ├── /screenshots
│   │   │   ├── TC001_step3_error.png
│   │   │   └── TC025_step5_error.png
│   │   └── summary_report.html
│   └── /2025-03-06
└── /2025-02
```

### 3.3 データ管理戦略

**テストデータの設計は保守性に直結する**

#### データの配置場所
```
/test_data
├── /master_data           【マスタデータ】
│   ├── users.csv          - テストユーザー一覧
│   ├── products.csv       - 商品マスタ
│   └── settings.json      - 環境設定
├── /test_cases            【テストケースデータ】
│   ├── login_patterns.csv
│   ├── search_patterns.csv
│   └── checkout_patterns.csv
└── /fixtures              【固定データ】
    ├── sample_image.png
    └── sample_pdf.pdf
```

#### データ形式の選択

| 形式 | 用途 | メリット | デメリット |
|------|------|---------|----------|
| **CSV** | テーブル形式のデータ | Excel編集可、軽量 | 階層構造に弱い |
| **JSON** | 設定ファイル、階層データ | 構造化しやすい | 人間には読みにくい |
| **YAML** | 設定ファイル | 人間に読みやすい | パース処理が必要 |
| **Excel** | 大量のテストパターン | 視認性が高い、関数使用可 | ファイルサイズ大 |

**使い分けの例：**
```
【ユーザー情報（CSV）】
user_id, email, password, role
1, user1@example.com, pass123, user
2, admin@example.com, adminpass, admin
3, guest@example.com, guestpass, guest

【環境設定（JSON）】
{
  "environments": {
    "dev": {
      "url": "https://dev.example.com",
      "timeout": 30
    },
    "staging": {
      "url": "https://staging.example.com",
      "timeout": 60
    }
  }
}

【テストパターン（Excel）】
- 複雑な境界値テスト
- 数式で自動計算
- 条件付き書式で視認性向上
```

#### データのバージョン管理
```
【データのバージョン管理戦略】

1. Git管理対象
   ✅ テストデータ（CSV, JSON）
   ✅ 設定ファイル
   ✅ 共通部品

2. Git管理対象外
   ❌ 実行結果（ログ、スクリーンショット）
   ❌ 大容量ファイル（動画、大きな画像）
   ❌ 環境依存の一時ファイル

3. ブランチ戦略
   - main: 本番環境用データ
   - develop: 開発環境用データ
   - feature/*: 機能別のテストデータ
```

## 4. CI/CD統合の設計

### 4.1 テスト実行戦略

**いつ、どのテストを実行するか？**

#### 実行タイミングの設計
```
【実行スケジュール】

1. コミット時（Commit）
   - 対象: スモークテスト（重要機能のみ、5分以内）
   - 頻度: コミットごと
   - 目的: 基本機能の即座確認

2. プルリクエスト時（PR）
   - 対象: 影響範囲のテスト（30分以内）
   - 頻度: PRごと
   - 目的: マージ前の品質ゲート

3. 夜間バッチ（Nightly）
   - 対象: 全テストスイート（2時間以内）
   - 頻度: 毎日深夜2時
   - 目的: 包括的な品質確認

4. リリース前（Pre-release）
   - 対象: リグレッションテスト全件（4時間以内）
   - 頻度: リリース前
   - 目的: 最終品質保証
```

**テストピラミッドの適用：**
```
        △ E2Eテスト（10%）
       △△△ 統合テスト（30%）
      △△△△△ 単体テスト（60%）
     
【実行時間の配分】
- 単体: 1分以内
- 統合: 5分以内
- E2E: 30分以内
```

#### テストの優先順位付け

**Priority設定の例：**

| Priority | テスト内容 | 実行頻度 | 実行時間目標 |
|----------|----------|---------|------------|
| **P0（最重要）** | ログイン、決済など | 毎コミット | 5分以内 |
| **P1（重要）** | 主要機能 | 毎PR | 30分以内 |
| **P2（通常）** | 補助機能 | 毎日 | 2時間以内 |
| **P3（低）** | エッジケース | 週1回 | 制限なし |

**タグ付けの例：**
```
テストケース名: LOGIN_正常系_標準ユーザー
タグ: [P0, smoke, authentication, daily]

テストケース名: SEARCH_境界値_特殊文字
タグ: [P2, boundary, search, weekly]

実行コマンド例：
- コミット時: タグ [P0] のみ実行
- PR時: タグ [P0, P1] を実行
- 夜間: タグ [P0, P1, P2] を実行
- 週次: 全テスト実行
```

### 4.2 並列実行の設計

**実行時間を短縮するための並列化戦略**

#### 並列実行の設計パターン
```
【シーケンシャル実行（従来）】
TC001 → TC002 → TC003 → TC004 → TC005
合計: 25分

【並列実行（5並列）】
TC001 ┐
TC002 ├─ 同時実行
TC003 │
TC004 │
TC005 ┘
合計: 5分（80%短縮！）
```

**並列実行の注意点：**

| 考慮事項 | 対策 |
|---------|------|
| **データの競合** | テストごとに独立したデータを使用 |
| **リソース不足** | 並列数を適切に制限（CPUコア数の2倍程度） |
| **依存関係** | 依存するテストはグループ化して順次実行 |
| **不安定性** | タイムアウト時間を長めに設定 |

**並列実行の設計例：**
```
【グループ分け】
Group A（並列実行可能）:
  - LOGIN系テスト（独立したユーザーを使用）
  - SEARCH系テスト（読み取り専用）
  - 表示系テスト（データ変更なし）

Group B（順次実行必要）:
  - データ投入テスト
  - データ削除テスト
  - トランザクション系テスト

実行戦略:
  1. Group A → 10並列で実行（5分）
  2. Group B → 順次実行（10分）
  合計15分（並列化なしなら50分）
```

## 7. よくある質問（FAQ）

### Q1：設計に時間をかけすぎて実装が遅れそうです...

**A1：** 「設計8割、実装2割」の原則で考えましょう。
```
【時間配分の目安】
全体100時間のプロジェクトの場合：

設計フェーズ（30時間）:
  - アーキテクチャ設計: 10時間
  - 共通部品設計: 10時間
  - 命名規則・規約策定: 5時間
  - レビュー: 5時間

実装フェーズ（50時間）:
  - 共通部品実装: 10時間
  - テストケース実装: 35時間
  - レビュー・修正: 5時間

安定化フェーズ（20時間）:
  - バグ修正: 10時間
  - ドキュメント整備: 5時間
  - CI/CD統合: 5時間

→ 最初の30%を設計に使うことで、
  後半70%がスムーズに進む！
```

**設計不足の場合のリスク：**
- 実装後に大幅な作り直し（2倍の時間）
- メンテナンス工数の増大（年間で数倍）
- チーム全体の生産性低下

### Q2：Page Objectパターンって必須ですか？

**A2：** 必須ではありませんが、100TC以上なら強く推奨します。
```
【TC数による判断】

20TC以下:
  → Page Objectなしでも管理可能
  → シンプルな設計でOK

50TC程度:
  → Page Object導入を検討
  → 頻繁に使う画面だけでも導入

100TC以上:
  → Page Object必須
  → ないとメンテナンス地獄

500TC以上:
  → フレームワーク導入を検討
  → 専任アーキテクトが必要
```

### Q3：共通部品が多すぎて逆に複雑になっています...

**A3：** 「過度な共通化」は設計の失敗パターンです。
```
【悪い例：何でも共通化】
common_click_button_with_retry_and_screenshot()
common_input_text_with_validation_and_clear()
common_wait_for_element_with_custom_timeout()

→ 名前が長い、使いにくい、保守困難

【良い例：適切な粒度】
common_login()
common_search()
common_add_to_cart()

→ 目的が明確、使いやすい、保守容易
```

**共通化の基準（再掲）：**
- 3回以上使われる
- 5ステップ以上ある
- 仕様変更の可能性がある

### Q4：新メンバーが設計を理解できません...

**A4：** ドキュメントとオンボーディングの充実が必要です。
```
【新メンバー向けオンボーディング資料】

1日目:
  - アーキテクチャ設計書を読む
  - フォルダ構成の説明を受ける
  - サンプルTCを実行してみる

2日目:
  - 既存TCを読んで理解する
  - 共通部品の使い方を学ぶ
  - 簡単なTCを1つ作成（レビュー付き）

3日目:
  - データ管理方針を学ぶ
  - 中程度のTCを作成
  - コーディング規約を確認

1週間後:
  - 独立してTC作成可能に
  - 設計レビューに参加

1ヶ月後:
  - 設計改善提案ができる
```

**ドキュメント整備のチェックリスト：**
- [ ] アーキテクチャ図がある
- [ ] フォルダ構成が説明されている
- [ ] 命名規則が明文化されている
- [ ] 共通部品の使い方が分かる
- [ ] サンプルコードがある
- [ ] FAQ/トラブルシューティングがある

> **アーキテクトの金言**
>
> 「今日動くコード」ではなく「3年後も保守できる設計」を。
> 
> 設計は最初の1週間で決まるが、
> その影響は3年間続く。

>
> 未来を見据えて設計し、
> 現在のチームを成功に導く。
> それがアーキテクトの仕事です。

1. **既存プロジェクトを見直す**

   - 共通部品化できる処理はないか？

   - どんな選択肢があったのか？
   - 次はどう改善するか？
